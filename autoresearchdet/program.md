# autoresearch-det — YOLO11 Fine-tuning

You are an autonomous ML researcher. Your goal: find the best fine-tuning
configuration for YOLO11 on a small custom dataset (starting from COCO
pretrained weights), maximising `val/mAP50-95` (primary) and `val/mAP50` (secondary).

## 10-Experiment Budget Strategy

You have exactly **10 experiments**. Use them wisely — do not waste budget on
low-confidence or redundant changes. Follow this 3-phase allocation:

### Phase 1 — Baseline (exp 01)
Run `train.py` as-is. This is your reference point. Never reset this commit.

### Phase 2 — Exploration: High-impact single changes (exp 02–06)
Pick changes from the priority table below (highest ROI first).
Change **one variable at a time** so you can isolate the effect.
If a change improves mAP → keep; if not → discard and try the next.

Priority order for small-dataset YOLO fine-tuning:
```
1. LR0          — most impactful; try 5e-4 or 2e-3 vs baseline 1e-3
2. OPTIMIZER    — AdamW vs SGD (SGD sometimes better on very small sets)
3. CLOSE_MOSAIC — try 20 (disable mosaic earlier); stabilises late training
4. MOSAIC       — try 0.5 or 0.0 if dataset < 200 images
5. LRF          — try 0.1 (less aggressive cosine decay)
6. WEIGHT_DECAY — try 5e-4 or 1e-3
7. WARMUP_EPOCHS— try 5.0 for more stable start
8. IMGSZ        — try 320 (faster) or 1280 (if VRAM allows)
9. MODEL_SIZE   — try "s" if baseline uses "n" and VRAM < 6 GB
10. LABEL_SMOOTHING — try 0.1
```

**Dataset-size calibration rules** (read nc and image count from data.yaml):
- < 100 images  → reduce MOSAIC to 0.3, MIXUP=0.0, increase EPOCHS to 80
- 100–500 images → MOSAIC=0.8, EPOCHS=50, consider COPY_PASTE=0.1
- > 500 images  → default augmentation is fine; focus on LR and optimizer

### Phase 3 — Exploitation: Combine & refine (exp 07–10)
By exp 07, you should have 2–4 "keep" changes. Now:
- **Combine** all kept changes into one config (exp 07 or 08).
- If combined is worse than individual bests, revert and try fine-grained tuning
  of the single best parameter found so far.
- Reserve exp 09–10 for: `MODEL_SIZE` bump ("n"→"s") or `IMGSZ` increase,
  only if VRAM headroom exists (check `peak_vram_mb` in results).

## Experiment Loop

LOOP until `results.tsv` has 10 data rows (excluding header):

```
1.  Read results.tsv → count completed experiments.
2.  Read current train.py.
3.  Analyse history: which changes improved mAP? Which were discarded?
4.  Select next action based on current phase (see Budget Strategy above).
5.  Edit train.py with the chosen change.
6.  git commit -am "experiment: <description>"
7.  python train.py > run.log 2>&1
8.  grep "^val_mAP5095:\|^val_mAP50:\|^peak_vram_mb:" run.log
9.  If empty → CRASH. Go to Crash Recovery below.
10. Compare val_mAP5095 to best so far:
      improved  → status=keep   (keep commit, advance branch)
      no change → status=discard (git reset --hard HEAD~1, restore previous train.py)
11. Append row to results.tsv.
12. python plot_progress.py
```

After 10 experiments, print Final Summary (see below).

---

## Crash Recovery

```
1. tail -n 80 run.log  → identify error type
2. Diagnose:
   - "CUDA out of memory"  → halve BATCH; if BATCH < 4 then skip this change
   - "NaN loss"            → halve LR0; set AMP=False if persists
   - "FileNotFoundError"   → check DATA_YAML path; run prepare.py --verify
   - import error          → pip install missing package, retry once
   - other                 → git reset --hard HEAD~1, skip this experiment
3. After fix: retry the same experiment once.
4. Second failure → log status=crash, git reset, move to next idea.
```

## Key Principles

- **Never** reset the baseline commit (exp 01).
- **One change at a time** in Phase 2 — never change 2+ params simultaneously
  unless you are in Phase 3 (combining confirmed-good changes).
- **Read VRAM** from results.tsv before upsizing model or imgsz.
- **mAP50-95 is primary**. Prefer a change that raises mAP50-95 by 0.002
  over one that raises only mAP50 by 0.01.
- If stuck (3 consecutive discards), jump to Phase 3 early.
